## 1. Executive Summary

The HELIX staking protocol is well-architected for Solana runtime constraints, with correct use of u128 intermediate math, robust duplicate-prevention in BPD distribution, and proper checked arithmetic throughout. The primary runtime risks center on **CU exhaustion at MAX_STAKES_PER_BPD=20 with full deserialization+PDA verification per account** (Agent 1), **absence of ComputeBudget and priority fee instructions in the frontend** (Agent 5), and **slot-to-day conversion drift over the 180-day claim period** (Agent 3). No critical fund-loss vulnerabilities were found.

## 3. Clean Areas (What's Secure)

| Area | Assessment |
|------|-----------|
| **Checked arithmetic** | All math uses `checked_add/sub/mul/div` or `mul_div()` with u128 intermediates. Overflow-checks enabled in release. No raw `as u64` truncation in business logic (only in event emission). |
| **Duplicate BPD prevention** | `bpd_claim_period_id` and `bpd_finalize_period_id` per-stake fields prevent double-counting across batches. `claim_period_id > 0` requirement (MED-5 fix) prevents collision with default 0. |
| **PDA verification in BPD loops** | Both `finalize_bpd_calculation` and `trigger_big_pay_day` verify PDA derivation for every remaining account. Owner check + discriminator check + PDA address match = robust forgery prevention. |
| **Reentrancy protection** | State mutations happen BEFORE CPIs throughout: `stake.is_active = false` before MintTo in `unstake`, `reward_debt` updated before mint in `claim_rewards`, `withdrawn_amount` updated before mint in `withdraw_vested`. |
| **BPD window locking** | `global_state.reserved[0]` BPD window flag correctly blocks `unstake` and authority transfer during BPD distribution, preventing share manipulation during calculation. |
| **Authority transfer** | Two-step transfer (propose + accept) with BPD window blocking. Clean pattern. |
| **Merkle proof verification** | Standard sorted-pair Merkle proof with keccak256 via `solana-nostd-keccak`. Leaf includes `claim_period_id` preventing cross-period replay. |
| **Ed25519 signature verification** | Proper instruction introspection checking the immediately preceding instruction. Message format includes pubkey and amount, preventing signature reuse. `claimer == snapshot_wallet` constraint prevents delegation attacks. |
| **Frontend math parity** | `app/web/lib/solana/math.ts` mirrors on-chain `instructions/math.rs` exactly: same formulas, same BN precision, same rounding direction (mulDivUp for penalties). |
| **Event emission** | All instructions emit events with `slot` field for reorg correlation. Comprehensive event coverage for indexer. |
| **Token-2022 usage** | Correct burn-and-mint model. Mint authority is PDA. `spl-token-2022` used correctly via `anchor-spl` interfaces. |

## 4. Final Verdict

### **CONDITIONAL PASS**

**Runtime Risk Score: 3.5 / 10** (Low-Medium)

**Conditions for PASS:**

1. **[Must Fix — RT-01, RT-02, RT-03]** Add explicit `ComputeBudgetProgram.setComputeUnitLimit()` and `setComputeUnitPrice()` to ALL frontend transaction construction, especially BPD batch operations. Without this, BPD distribution **will intermittently fail on mainnet** with 20 remaining accounts due to CU exhaustion.

2. **[Must Fix — RT-13]** Produce verifiable build before mainnet deployment. Run `anchor build --verifiable` and publish the hash.

3. **[Should Fix — RT-06]** Refactor double-deserialization in `trigger_big_pay_day` to reduce CU consumption by ~40-60k per 20-account batch. This provides critical CU headroom.

4. **[Should Fix — RT-14]** Set mainnet program ID in Anchor.toml.

**Assessment rationale:** The on-chain program logic is sound — arithmetic is safe, access control is correct, BPD distribution handles edge cases properly, and the three-phase BPD flow (finalize → seal → trigger) is well-designed. The gaps are operational (CU budgeting, priority fees, verifiable builds) rather than logical. These are addressable without program changes — they require frontend/tooling updates and deployment process improvements.

**Risk Breakdown:**
- Fund loss risk: **Negligible** — no paths to drain funds, steal rewards, or bypass access control
- Liveness risk: **Medium** — BPD transactions may fail without CU budget; congestion may delay time-sensitive operations without priority fees  
- Accuracy risk: **Low** — slot-to-day drift is documented and mitigable via `admin_set_slots_per_day`
- Deployment risk: **Low** — binary size healthy, no verifiable build yet but straightforward to produce
