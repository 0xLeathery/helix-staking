## 2. Duration Loyalty Multiplier — Design Research

### 2.1 Problem Statement

Currently, daily inflation rewards are distributed proportionally to T-shares. A stake's T-shares are calculated **once at creation** using LPB (Longer Pays Better) and BPB (Bigger Pays Better) bonuses. Once staked, there is no additional incentive to remain staked — a 365-day stake earns the same daily rate on day 1 as on day 364.

This creates a **weak hold incentive**: rational actors may unstake as soon as favorable conditions change, since the opportunity cost of staying staked doesn't decrease over time.

### 2.2 HEX Reference

Original HEX on Ethereum does NOT have a duration loyalty multiplier. T-shares are fixed at creation. However, HEX relies on:
- The penalty system (50%+ early unstake) as the hold incentive
- Share rate increase making future restaking more expensive
- Social/narrative pressure ("staking is locking, don't break your CD")

For HELIX on Solana, adding a loyalty multiplier is a **protocol improvement** over HEX, not a departure.

### 2.3 Design Options

**Option A: Time-Weighted Effective T-Shares (Recommended)**

Modify the `crank_distribution` reward calculation to weight each stake's T-shares by the proportion of its committed term already served:

```
effective_t_shares = t_shares × (1 + loyalty_bonus)
loyalty_bonus = min(days_served / committed_days, 1.0) × LOYALTY_MAX_BONUS
```

Where `LOYALTY_MAX_BONUS = 0.5` (50% bonus at full term).

- Day 1 of a 365-day stake: loyalty = 0%, effective = 1.0x
- Day 182 of a 365-day stake: loyalty = 25%, effective = 1.25x
- Day 365 of a 365-day stake: loyalty = 50%, effective = 1.5x

**Pros:**
- Simple linear curve, easy to audit
- Retroactive — existing stakes immediately benefit proportional to time served
- Doesn't change T-share accounting, only reward distribution weighting
- Rewards patience without punishing new stakers

**Cons:**
- Modifies `crank_distribution` (performance-sensitive batch instruction)
- Needs `StakeAccount` reads during crank (currently crank only reads GlobalState)

**Option B: Accrual Rate Tiers**

Define discrete tiers based on time served:
- 0-90 days: 1.0x reward rate
- 91-365 days: 1.15x reward rate
- 366-1000 days: 1.3x reward rate
- 1001+ days: 1.5x reward rate

**Pros:** Simple to implement, no continuous calculation
**Cons:** Cliff effects are gameable; doesn't scale smoothly

**Option C: Boost on Claim (Lazy Calculation)**

Instead of modifying crank, apply the loyalty multiplier when the user calls `claim_rewards`. The share_rate-based reward is multiplied by the loyalty factor at claim time.

```
claim_amount = pending_rewards × (1 + loyalty_bonus)
loyalty_bonus = min(days_served / committed_days, 1.0) × LOYALTY_MAX_BONUS
```

**Pros:**
- Zero change to `crank_distribution` — no batch compute impact
- Lazy calculation matches existing architecture (lazy reward distribution)
- Only affects the claiming user's transaction
- Simple to test — `claim_rewards` already reads StakeAccount with all needed fields

**Cons:**
- Loyalty bonus is not reflected in real-time reward displays until claim
- "Pending rewards" shown on dashboard may differ from actual claim amount

**Recommendation: Option C (Boost on Claim)**

This matches the existing lazy distribution architecture. The crank updates `share_rate` globally, and individual stakes compute their rewards at claim time. Adding a loyalty multiplier at claim time is the minimal-disruption path. The frontend can pre-compute and display the loyalty-adjusted amount.

### 2.4 Implementation Details (Option C)

**On-chain changes:**
- `claim_rewards.rs`: After calculating `pending_rewards`, apply loyalty multiplier
- `unstake.rs`: Same loyalty multiplier applies to auto-claimed rewards during unstake
- `constants.rs`: Add `LOYALTY_MAX_BONUS: u64 = 500_000_000` (0.5x = 50% in PRECISION units)
- `math.rs`: Add `calculate_loyalty_bonus(days_served: u64, committed_days: u64) -> Result<u64>`

**Frontend changes:**
- `useStakes` hook: Calculate and display loyalty-adjusted pending rewards
- Math module: Add `calculateLoyaltyBonus` matching on-chain formula
- Penalty calculator: Show loyalty multiplier in stake details

**StakeAccount fields already available:**
- `start_slot` — when stake was created
- `stake_days` — committed duration
- `clock.slot` — current slot
- `global_state.slots_per_day` — for days calculation

No account expansion needed.

## 3. Anti-Whale Mechanics — Design Research

### 3.1 Problem Statement

Two whale attack surfaces:
1. **BPD dominance**: A whale with massive T-share-days captures the majority of unclaimed tokens
2. **Daily reward dominance**: Large stakes earn proportionally more inflation, with no diminishing returns

### 3.2 Design Options

**Option A: BPD Share Cap (Recommended for BPD)**

Cap the maximum BPD any single stake can receive at a percentage of the total unclaimed pool:

```
max_bpd_per_stake = unclaimed_pool × BPD_WHALE_CAP_PCT / 100
actual_bpd = min(calculated_bpd, max_bpd_per_stake)
```

Where `BPD_WHALE_CAP_PCT = 5` (no single stake can take more than 5% of the BPD pool).

Excess tokens remain in the unclaimed pool and get redistributed proportionally to remaining stakes.

**Pros:**
- Straightforward implementation in `trigger_big_pay_day`
- Single constant to tune
- Excess redistributed fairly

**Cons:**
- Whales can split across multiple stakes to bypass (Sybil)
- Need per-*user* cap, not per-stake cap, but user identity is just a pubkey (also Sybil-able)

**Mitigation for Sybil:** Cap per-user across all stakes by summing BPD distributed per user wallet. This requires tracking in `trigger_big_pay_day` remaining accounts.

**Practical approach:** Per-stake cap is the pragmatic choice. It raises the cost of whale extraction (must manage N stakes each below cap) without being Sybil-proof. Perfect is the enemy of good.

**Option B: BPB Diminishing Returns (Recommended for Daily Rewards)**

Currently BPB (Bigger Pays Better) is linear: `bonus = amount * PRECISION / BPB_THRESHOLD` capped at 1x. This means staking 150M tokens gets exactly 1x bonus = 2x base. Beyond 150M, no additional bonus but no penalty either.

Add a **logarithmic diminishing return** above the BPB threshold:

```
if amount <= BPB_THRESHOLD:
    bonus = amount * PRECISION / BPB_THRESHOLD  // linear 0-1x (unchanged)
else:
    base = PRECISION  // 1x at threshold
    excess = amount - BPB_THRESHOLD
    diminishing = ln(1 + excess / BPB_THRESHOLD) * PRECISION / ln(2)
    bonus = base + min(diminishing, PRECISION / 2)  // caps at 1.5x total
```

**Challenge:** Logarithmic functions on-chain are expensive and impractical in Solana BPF. Alternative — **piecewise linear with decreasing slope:**

```
if amount <= TIER_1 (150M):     bonus = amount / TIER_1               // 0 → 1.0x
elif amount <= TIER_2 (500M):   bonus = 1.0 + (amount - TIER_1) / (TIER_2 - TIER_1) * 0.25  // 1.0 → 1.25x
elif amount <= TIER_3 (1B):     bonus = 1.25 + (amount - TIER_2) / (TIER_3 - TIER_2) * 0.15  // 1.25 → 1.4x
else:                           bonus = 1.5  // hard cap
```

**Pros:**
- Integer arithmetic, no floating point or logarithms
- Whales still benefit from larger stakes, but with diminishing returns
- Preserves existing behavior for stakes ≤ 150M (backward compatible)
- Adjustable tiers via constants

**Cons:**
- More complex math module
- Existing T-shares already calculated for active stakes (not retroactive unless we add a multiplier at distribution)

**Option C: Hard Cap on Stake Size**

Simply reject stakes above a maximum amount (e.g., 500M HELIX).

**Rejected:** Too blunt. Already Sybil-bypassable. Better to use economic incentives.

### 3.3 Recommendation

Combine **BPD share cap (5% per stake)** and **BPB diminishing returns** (piecewise linear above 150M threshold). Both can be implemented as constant-driven math changes:

1. **BPD cap**: Modify `trigger_big_pay_day` to enforce `max_bpd_per_stake`
2. **BPB diminishing returns**: Modify `calculate_bpb_bonus` in `math.rs` with tiered calculation
3. **Excess BPD redistribution**: Uncapped remainder stays in pool, gets redistributed to remaining stakes proportionally in subsequent batches

### 3.4 Implementation Details

**BPD Cap (trigger_big_pay_day.rs):**
- After calculating `bonus`, cap at `bpd_remaining_unclaimed * BPD_MAX_SHARE_PER_STAKE / 100`
- New constant: `BPD_MAX_SHARE_PER_STAKE_PCT: u64 = 5`
- Excess stays in `bpd_remaining_unclaimed` for remaining distributions

**BPB Diminishing Returns (math.rs):**
- Modify `calculate_bpb_bonus` to use tiered calculation above `BPB_THRESHOLD`
- New constants: `BPB_TIER_2`, `BPB_TIER_3`, `BPB_MAX_BONUS`
- Only affects new stakes (existing T-shares unchanged)
- This is an **economic deterrent** — whales get diminishing returns on NEW stakes

**No account expansion needed** for either mechanic.

## 4. BPD Transparency & Completeness Verification

### 4.1 Problem

Authority controls BPD finalization. Community cannot verify completeness on-chain. The security audit recommends transparency events and a governance delay.

### 4.2 Recommendation

1. **Batch progress events** from `finalize_bpd_calculation`:
   ```rust
   emit!(BpdBatchFinalized {
       claim_period_id,
       batch_stakes_processed,
       total_stakes_finalized,
       cumulative_share_days,
       timestamp,
   });
   ```

2. **Seal delay constraint**: `seal_bpd_finalize` requires `clock.unix_timestamp >= finalize_start_timestamp + SEAL_DELAY_SECONDS` (24-48h delay between last finalize batch and seal).

3. **Permissionless verification view**: Off-chain — indexer tracks finalized stakes, any user can check if their stake was included by comparing `bpd_finalize_period_id == claim_period_id`.

### 4.3 Implementation

- New event struct: `BpdBatchFinalized`
- New field in ClaimConfig: `bpd_finalize_start_timestamp` (set on first batch)
- New constant: `BPD_SEAL_DELAY_SECONDS: i64 = 86400` (24 hours)
- Constraint in `seal_bpd_finalize`: verify delay elapsed
- Indexer: new event handler for BpdBatchFinalized, new API endpoint for stake finalization status
