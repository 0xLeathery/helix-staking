## Tasks

### Task 1: Create ROLLBACK.md (OPS-12)

**File:** `.planning/ROLLBACK.md` (NEW)

Document:
1. **Pre-deployment preparation:**
   - Always keep the previous `.so` binary in version control (tag each release)
   - Create a buffer account BEFORE deploying any upgrade
   - Test rollback procedure on devnet before mainnet

2. **Emergency rollback procedure:**
   ```bash
   # Step 1: Stage the previous program binary in a buffer
   solana program write-buffer ./target/deploy/helix_staking-v{PREVIOUS}.so

   # Step 2: Verify buffer contents
   solana program show --buffers

   # Step 3: Deploy from buffer (requires upgrade authority)
   solana program deploy --buffer <BUFFER_ADDRESS> --program-id <PROGRAM_ID>

   # For multisig: use Squads to propose the buffer deploy
   # All multisig members must approve within the configured timeframe
   ```

3. **Verification after rollback:**
   - Compare on-chain bytecode hash with known-good hash from version control
   - Run smoke test against the rolled-back program
   - Verify indexer is processing events correctly
   - Check frontend can still construct valid transactions

4. **When to rollback vs pause:**
   - **Pause first** if the vulnerability allows continued fund safety while paused
   - **Rollback immediately** if the deployed code has a fund-draining vulnerability
   - **Never rollback** during an active BPD window (abort BPD first)
