**Accounts struct changes:**
Add a new constraint to `claim_config`:
- `constraint = claim_config.bpd_calculation_complete @ HelixError::BpdCalculationNotComplete`

This replaces the need for the function to calculate rate -- it gates entry on having a pre-calculated rate.

**Function rewrite -- key changes from current code:**

1. **Remove first-batch detection and unclaimed calculation** (lines 49-66 current code). The unclaimed amount and share-days are already calculated by finalize_bpd_calculation and stored in ClaimConfig. Load the pre-calculated rate directly:
   ```rust
   let helix_per_share_day = claim_config.bpd_helix_per_share_day;
   ```
   If `helix_per_share_day == 0`, mark complete and return (nothing to distribute).

2. **Keep remaining_accounts iteration** (lines 88-163 current code) but with these modifications:
   - After deserializing each stake, ADD duplicate prevention check BEFORE eligibility checks:
     ```rust
     if stake.bpd_claim_period_id == claim_config.claim_period_id {
         continue; // Already received BPD this period
     }
     ```
   - Keep all existing eligibility checks (is_active, start_slot range, days_staked > 0)
   - Keep share_days calculation

3. **Remove the per-batch rate calculation block** (lines 217-224 current code). Use the pre-calculated `helix_per_share_day` from step 1 instead.

4. **In the distribution loop** (lines 229-256 current code), after updating `stake.bpd_bonus_pending`:
   - ADD: `stake.bpd_claim_period_id = claim_config.claim_period_id;` (mark as received AFTER bonus set, BEFORE serialize)
   - The serialize call `stake.try_serialize(...)` already exists and will persist both fields

5. **Remove the `batch_eligible_share_days` accumulation** into `claim_config.bpd_total_share_days` (line 213-215 current code). Share-days are already fully accumulated by finalize_bpd_calculation. Don't re-add them.

6. **Keep the "is last batch" detection and completion logic** (lines 271-286 current code) but simplify: mark complete when `ctx.remaining_accounts.len() < MAX_STAKES_PER_BPD` (indicates final batch).

7. **Keep the BigPayDayDistributed event emission** (lines 289-297 current code) but update `total_eligible_share_days` to read from `claim_config.bpd_total_share_days` (already global) and `helix_per_share_day` from the pre-calculated value.

8. **Handle the empty-batch-first-call case** differently now: Since finalize is already complete, if no stakes are passed or all are filtered out, just mark complete and return. The "keep pool pending" logic is no longer needed in trigger_big_pay_day because finalize_bpd_calculation handles the "no eligible stakes" case.

**Summary of what's removed:**
- First-batch unclaimed amount calculation (now in finalize)

- bpd_total_share_days accumulation in trigger (now done in finalize)

**Step 1: Add helper function for finalize_bpd_calculation**

Add a helper at the top of the describe block (after `setupClaimPeriodWithStaker`):

```typescript
async function finalizeBpd(
  program: any,
  payer: any,
  globalState: any,
  claimConfigPDA: any,
  stakePDAs: any[],
) {
  const remainingAccounts = stakePDAs.map(pubkey => ({
    pubkey, isSigner: false, isWritable: false,
  }));
  await program.methods
    .finalizeBpdCalculation()
    .accounts({
      caller: payer.publicKey,
      globalState,
      claimConfig: claimConfigPDA,
    })
    .remainingAccounts(remainingAccounts)
    .signers([payer])
    .rpc();
}
```

Note: remaining_accounts for finalize are NOT writable (read-only scan). For trigger they remain writable.
