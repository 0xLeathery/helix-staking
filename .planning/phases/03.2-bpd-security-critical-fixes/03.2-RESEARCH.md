# Phase 3.2: BPD Security Critical Fixes - Research

**Researched:** 2026-02-08
**Domain:** Solana program security, batch distribution patterns, duplicate prevention mechanisms
**Confidence:** HIGH

## Summary

Phase 3.2 addresses two CRITICAL security vulnerabilities in the Big Pay Day (BPD) distribution mechanism introduced in Phase 3.1's pagination implementation:

1. **CRIT-1**: BPD rate calculated per-batch instead of globally, allowing first batch to drain entire pool
2. **CRIT-2**: No duplicate prevention mechanism allows same stake to receive BPD multiple times

The research confirms these are fundamental design flaws in the pagination implementation that enable complete protocol fund drainage. The fix requires implementing duplicate prevention via per-period tracking and ensuring rate calculation uses global share-days aggregated across all eligible stakes.

**Primary recommendation:** Implement Option A (Two-Phase Distribution) with `bpd_claim_period_id` field added to StakeAccount for duplicate prevention. This maintains the existing pagination pattern while fixing the critical vulnerabilities without breaking the 109-byte StakeAccount layout or existing tests.

## Architecture Patterns

### Current BPD Distribution Architecture (FLAWED)

```
┌─────────────────────────────────────────────────────────┐
│ trigger_big_pay_day (current - VULNERABLE)             │
├─────────────────────────────────────────────────────────┤
│ 1. Calculate unclaimed amount (first batch only)       │
│ 2. Iterate THIS BATCH's remaining_accounts (max 20)    │
│ 3. Calculate batch_eligible_share_days (ONLY this batch)│
│ 4. Calculate rate = unclaimed / batch_share_days   ← BUG│
│ 5. Distribute to batch stakes                          │
│ 6. Update bpd_remaining_unclaimed                      │
│ 7. If more stakes, call again with next batch          │
└─────────────────────────────────────────────────────────┘

Problem: Each batch calculates rate independently
Result: First batch gets disproportionate share
```

**Attack Scenario (CRIT-1)**:
```
Pool: 1,000,000 HELIX unclaimed

Batch 1 (attacker): 1000 share-days
  rate = 1,000,000 / 1000 = 1000 HELIX/share-day
  receives: 1,000,000 HELIX (entire pool)
  remaining: 0

Batch 2+ (legitimate): 9000 share-days
  rate = 0 / 9000 = 0
  receives: 0 HELIX
```

**Attack Scenario (CRIT-2)**:
```
Same transaction:
  remaining_accounts = [stake_A, stake_A, stake_A, ...stake_A] (20x)
  Each iteration: stake_A.bpd_bonus_pending += bonus
  Result: 20x the intended bonus

Across batches:
  Batch 1: Include stake_A → receives bonus
  Batch 2: Include stake_A again → receives bonus again
  No tracking prevents this
```

### Recommended Architecture: Two-Phase Distribution

```
┌─────────────────────────────────────────────────────────┐
│ Phase 1: finalize_bpd_calculation (NEW)                │
├─────────────────────────────────────────────────────────┤
│ 1. Verify claim period ended                           │
│ 2. Iterate ALL stakes (paginated, read-only)           │
│ 3. Accumulate global_eligible_share_days                │
│ 4. Calculate helix_per_share_day = pool / global_share_days│
│ 5. Store in ClaimConfig.bpd_helix_per_share_day (NEW)  │
│ 6. Set ClaimConfig.bpd_calculation_complete (NEW)      │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ Phase 2: trigger_big_pay_day (MODIFIED)                │
├─────────────────────────────────────────────────────────┤
│ 1. Require bpd_calculation_complete = true             │
│ 2. Load pre-calculated helix_per_share_day             │
│ 3. Iterate batch stakes (remaining_accounts, max 20)   │
│ 4. For each stake:                                      │
│    a. CHECK: stake.bpd_claim_period_id != current   ← FIX│
│    b. Calculate: bonus = share_days × rate             │
│    c. Update: stake.bpd_bonus_pending += bonus         │
│    d. SET: stake.bpd_claim_period_id = current_period ← FIX│
│ 5. Track progress, continue if more batches needed     │
└─────────────────────────────────────────────────────────┘
```

### Pattern 1: Duplicate Prevention via Per-Period Tracking

**What:** Each StakeAccount tracks which claim period it last received BPD for
**When to use:** Any multi-batch distribution where same account could appear multiple times
**Why it works:** O(1) check per stake, no additional account lookups

```rust
// In StakeAccount (add field):
pub bpd_claim_period_id: u32,  // Last period this stake received BPD

// In trigger_big_pay_day (check before distribution):
if stake.bpd_claim_period_id == claim_config.claim_period_id {
    continue; // Already received BPD this period
}

// After successful distribution:
stake.bpd_claim_period_id = claim_config.claim_period_id;
```

**Tradeoffs:**
- ✅ Simple, efficient (no bitmap, no separate tracker account)
- ✅ Works across multiple batches and transactions
- ✅ No additional account rent
- ⚠️ Requires StakeAccount realloc (but migration already implemented)

### Pattern 2: Global Rate Calculation Before Distribution

**What:** Pre-calculate distribution rate using ALL eligible stakes before any distribution
**When to use:** When batch processing must maintain fairness across batches
**Why it works:** Decouples rate calculation from batch iteration

```rust
// ClaimConfig gains fields:
pub bpd_helix_per_share_day: u128,  // Pre-calculated rate
pub bpd_calculation_complete: bool,  // Phase gate

// finalize_bpd_calculation instruction:
let mut global_share_days: u128 = 0;
for stake in all_eligible_stakes_paginated {
    global_share_days += stake_share_days;
}
claim_config.bpd_helix_per_share_day =
    (unclaimed * PRECISION) / global_share_days;
claim_config.bpd_calculation_complete = true;

// trigger_big_pay_day uses pre-calculated rate:
require!(claim_config.bpd_calculation_complete);
let rate = claim_config.bpd_helix_per_share_day;
```

**Tradeoffs:**
- ✅ Guarantees fair distribution across batches
- ✅ Clear separation of concerns (calculate vs distribute)
- ⚠️ Requires new instruction (finalize_bpd_calculation)
- ⚠️ Adds workflow step (must call finalize before trigger)

### Pattern 3: StakeAccount Layout Migration (Already Implemented)

The project already uses safe account expansion via `realloc` with `zero=true`:

```rust
// In claim_rewards.rs (existing pattern):
#[account(
    mut,
    seeds = [STAKE_SEED, user.key().as_ref(), &stake_account.stake_id.to_le_bytes()],
    bump = stake_account.bump,
    realloc = StakeAccount::LEN,
    realloc::payer = user,
    realloc::zero = true,  // ← New fields default to 0/false
)]
pub stake_account: Account<'info, StakeAccount>,
```

**Migration characteristics:**
- OLD_LEN = 92 bytes (original Phase 2)
- CURRENT_LEN = 109 bytes (Phase 3: added bpd_bonus_pending, bpd_eligible, claim_period_start_slot)
- PROPOSED_LEN = 113 bytes (Phase 3.2: add bpd_claim_period_id: u32)
- Migration happens lazily on first claim_rewards call per stake
- `realloc::zero = true` ensures new u32 field defaults to 0 (safe: period IDs start at 1)

### Anti-Patterns to Avoid

1. **Bitmap Tracking**: Using a separate account with bitmap of processed stake IDs
   - Why bad: Additional account rent, complex indexing, limits to fixed size
   - Better: Per-account tracking with u32 field

2. **In-Transaction Duplicate Detection**: Tracking seen stakes within single transaction
   - Why bad: Doesn't prevent cross-batch attacks, complex state management
   - Better: Persistent per-period tracking

3. **Merkle-Based Distribution** (Option B from audit): Like free_claim pattern
   - Why bad: Requires off-chain calculation, changes user flow (users claim vs crank distributes)
   - When acceptable: If wanting to move to user-initiated claims instead of permissionless crank
   - Not recommended: Breaks existing design pattern (BPD is crank-triggered, not user-claimed)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Account size detection | Manual length checks | `StakeAccount::OLD_LEN` constant | Already implemented, tested |
| Safe realloc | Custom migration logic | `realloc::zero = true` | Anchor handles initialization |
| PDA validation | String parsing | `Pubkey::create_program_address` | Built-in in trigger_big_pay_day.rs:113-122 |
| Overflow-safe math | Custom SafeMath | Existing `checked_*` methods + u128 intermediates | Already standardized in math.rs |
| Rate precision | Custom fixed-point | `PRECISION = 1e9` constant | Project standard |

**Key insight:** The project already has robust patterns for account migration, overflow protection, and fixed-point math. Phase 3.2 fixes are additive, not replacing existing infrastructure.

## Common Pitfalls

### Pitfall 1: Breaking StakeAccount Layout Migration
**What goes wrong:** Changing field order or removing fields breaks existing stakes
**Why it happens:** Old accounts (92 bytes) get deserialized with new layout (113 bytes)
**How to avoid:**
- Only append new fields at end
- Never remove or reorder fields
- Keep OLD_LEN constant for detection
- Test migration path explicitly
**Warning signs:** Deserialization errors on old stakes, incorrect field values after realloc

### Pitfall 2: Off-By-One in Claim Period Comparison
**What goes wrong:** Using `!=` instead of `==` or `>=` in duplicate check
**Why it happens:** Confusion about whether period_id increments or resets
**How to avoid:**
```rust
// CORRECT: Check if already received THIS period
if stake.bpd_claim_period_id == claim_config.claim_period_id {
    continue; // Skip - already received
}

// WRONG: Check if different periods
if stake.bpd_claim_period_id != claim_config.claim_period_id {
    // This would ALLOW duplicates within same period!
}
```
**Warning signs:** Test with duplicate stakes in same batch passes when it should fail

### Pitfall 3: Forgetting to Set Duplicate Flag After Distribution
**What goes wrong:** Check passes but field never updated, allowing re-entry
**Why it happens:** Early returns or error paths skip the update
**How to avoid:**
```rust
// CORRECT: Set flag after successful distribution
stake.bpd_bonus_pending = stake.bpd_bonus_pending.checked_add(bonus)?;
stake.bpd_claim_period_id = claim_config.claim_period_id; // ← Must set!
stake.try_serialize(...)?;

// WRONG: Set flag before operations that might fail
stake.bpd_claim_period_id = claim_config.claim_period_id; // ← Too early!
stake.bpd_bonus_pending = stake.bpd_bonus_pending.checked_add(bonus)?; // ← Could fail
```
**Warning signs:** Duplicate attacks still work after fix applied

### Pitfall 4: Rate Calculation Overflow with Large Pools
**What goes wrong:** `(unclaimed * PRECISION) / global_share_days` overflows u128
**Why it happens:** unclaimed (u64) × PRECISION (u64 = 1e9) needs u128, but result might exceed
**How to avoid:**
```rust
// Use u128 for entire calculation:
let helix_per_share_day = (unclaimed_amount as u128)
    .checked_mul(PRECISION as u128)  // u128 × u128 = u128
    .ok_or(HelixError::Overflow)?
    .checked_div(batch_eligible_share_days)  // u128 ÷ u128 = u128
    .ok_or(HelixError::DivisionByZero)?;

// Store as u128 in ClaimConfig (not u64):
pub bpd_helix_per_share_day: u128,  // NOT u64!
```
**Warning signs:** Distribution fails with "Overflow" on large unclaimed amounts

### Pitfall 5: Not Testing Cross-Batch Duplicate Attacks
**What goes wrong:** Test single-batch duplicates but miss cross-batch scenario
**Why it happens:** Tests call trigger_big_pay_day once, not multiple times
**How to avoid:**
```typescript
// Test 1: Same stake twice in one batch
await triggerBigPayDay({
  remainingAccounts: [stakeA, stakeA]  // ← Should reject second
});

// Test 2: Same stake across batches
await triggerBigPayDay({
  remainingAccounts: [stakeA, stakeB, ...]  // Batch 1
});
await triggerBigPayDay({
  remainingAccounts: [stakeA, stakeC, ...]  // Batch 2 - stakeA again!
});
// stakeA should not receive bonus twice
```
**Warning signs:** Single-batch tests pass but vulnerability remains

### Pitfall 6: Unused Field Cleanup Breaking Existing Stakes
**What goes wrong:** Removing `bpd_eligible` and `claim_period_start_slot` breaks deserialization
**Why it happens:** Old accounts have these fields, removing them changes byte layout
**How to avoid:**
- Mark fields as deprecated in comments but keep in struct
- Or keep fields and start using them (e.g., use `bpd_eligible` in trigger_big_pay_day)
- Test that old accounts (109 bytes) still deserialize correctly
**Warning signs:** "AccountDidNotDeserialize" errors after deployment

### Example 2: Duplicate Prevention in trigger_big_pay_day

```rust
// File: programs/helix-staking/src/instructions/trigger_big_pay_day.rs

for (idx, share_days) in eligible_stakes.iter() {
    let account_info = &ctx.remaining_accounts[*idx];

    // Deserialize stake
    let mut stake: StakeAccount = StakeAccount::try_deserialize(
        &mut &account_info.try_borrow_data()?[..]
    )?;

    // === DUPLICATE PREVENTION CHECK ===
    if stake.bpd_claim_period_id == claim_config.claim_period_id {
        continue; // Already received BPD this period
    }

    // Calculate bonus
    let bonus = ((*share_days)
        .checked_mul(helix_per_share_day)
        .ok_or(HelixError::Overflow)?
        .checked_div(PRECISION as u128)
        .ok_or(HelixError::DivisionByZero)?) as u64;

    if bonus == 0 {
        continue;
    }

    // Update stake
    stake.bpd_bonus_pending = stake.bpd_bonus_pending
        .checked_add(bonus)
        .ok_or(HelixError::Overflow)?;

    // === MARK AS PROCESSED ===
    stake.bpd_claim_period_id = claim_config.claim_period_id;

    // Serialize back
    stake.try_serialize(&mut &mut account_info.try_borrow_mut_data()?[..])?;

    batch_distributed = batch_distributed
        .checked_add(bonus)
        .ok_or(HelixError::Overflow)?;
}
```

## State of the Art

### Solana Batch Processing Patterns (2026)

| Approach | When Used | Tradeoffs |
|----------|-----------|-----------|
| **remaining_accounts pagination** | Current helix-staking pattern | ✅ Permissionless ✅ No state bloat ⚠️ Needs duplicate prevention |
| **Session accounts** | Newer pattern (Squads v4) | ✅ Built-in progress tracking ⚠️ Requires cleanup ⚠️ Rent costs |
| **Merkle distribution** | Compressed airdrops, free_claim | ✅ No batching needed ⚠️ User-initiated only |
| **Indexed lookup tables** | Very large datasets (10k+) | ✅ Scales to millions ⚠️ Complex setup ⚠️ V0 tx only |

**Current best practice (2026):** For permissionless crank distributions under 1000 accounts, `remaining_accounts` with per-account duplicate tracking is standard. Session accounts and lookup tables are overkill for this scale.

### Account Realloc Safety (Anchor 0.30+)

```rust
// Modern pattern (used in helix-staking):
#[account(
    mut,
    realloc = NewSize::LEN,
    realloc::payer = payer,
    realloc::zero = true,  // ← Key: New space initialized to zero
)]
```

**Deprecated/outdated:**
- Manual `AccountInfo::realloc` (Anchor 0.29 and earlier)
- `realloc::zero = false` (unsafe for adding fields)
- Separate migration instructions (lazy migration preferred)

**Impact:** Phase 3.2 can safely add `bpd_claim_period_id: u32` field. With `realloc::zero = true`, old accounts get field initialized to 0 automatically, and since claim_period_id starts at 1, zero means "never received BPD" (safe default).

## Open Questions

### Question 1: Should We Use bpd_eligible Flag or Remove It?

**What we know:**
- `bpd_eligible` is set in create_stake.rs (line 136) based on claim period status
- `claim_period_start_slot` is set alongside it (line 137)
- Neither field is checked in trigger_big_pay_day.rs (uses slot-based validation instead)
- Audit finding HIGH-1 flags this as unused field

**What's unclear:**
- Original intent: Was this for optimization (skip ineligible stakes early)?
- Or was it meant as the authoritative eligibility flag?

**Recommendation:**
- **Option A (Use the flag):** Add check in trigger_big_pay_day: `if !stake.bpd_eligible { continue; }`
  - Pro: Respects original design, potentially cheaper (no slot math)
  - Con: Two sources of truth (flag vs slot validation)
- **Option B (Remove the flag):** Mark as deprecated but keep in struct for layout compatibility
  - Pro: Single source of truth (slot-based validation)
  - Con: Wastes 9 bytes per stake (bool + u64)
- **Chosen:** Option A - use the flag. It was set for a reason, and checking it early is cheaper than slot arithmetic.

### Question 2: ClaimConfig Realloc Strategy

**What we know:**
- ClaimConfig is singleton PDA (one per program)
- Phase 3.1 already expanded it (added bpd_remaining_unclaimed, bpd_total_share_days)
- Phase 3.2 needs to add bpd_helix_per_share_day (u128) and bpd_calculation_complete (bool)
- No existing ClaimConfig migration pattern in codebase

**What's unclear:**
- Should ClaimConfig be realloc'd in finalize_bpd_calculation instruction?
- Or should it be a one-time migration instruction?
- Or should fields be added with unused space from start?

**Recommendation:**
- ClaimConfig is created fresh each claim period via initialize_claim_period
- No migration needed - just update the `impl ClaimConfig::LEN` calculation
- Next claim period (period 2+) will use new layout automatically
- Current period (period 1) can still use old flow if BPD already triggered
- **Chosen:** No realloc needed. ClaimConfig is recreated per period, so just update LEN constant.

### Question 3: Compute Budget for finalize_bpd_calculation

**What we know:**
- Must iterate ALL eligible stakes to calculate global_share_days
- Current MAX_STAKES_PER_BPD = 20 for distribution (conservative)
- finalize_bpd_calculation only reads stakes (no writes), cheaper than trigger_big_pay_day

**What's unclear:**
- Can we process 50-100 stakes per call (read-only)?
- Or should we match MAX_STAKES_PER_BPD = 20 for consistency?
- What's the actual compute cost of read-only iteration?

**Recommendation:**
- Start conservative: MAX_STAKES_PER_FINALIZE = 20 (same as distribution)
- Makes pagination logic consistent between finalize and trigger
- Can be increased later if compute headroom allows
- **Chosen:** MAX_STAKES_PER_FINALIZE = 20, allow pagination if >20 eligible stakes exist.
