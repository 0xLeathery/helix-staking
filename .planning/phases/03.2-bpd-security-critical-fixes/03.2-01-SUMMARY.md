- `programs/helix-staking/src/lib.rs`

**Implementation:**
1. **New Instruction: `finalize_bpd_calculation`**
   - Permissionless, callable after claim period ends
   - Processes up to MAX_STAKES_PER_FINALIZE (20) stakes per call
   - READ-ONLY: Only reads stake data, never writes to StakeAccount
   - Accumulates global share-days across paginated batches
   - Calculates and stores `helix_per_share_day` in ClaimConfig once all stakes processed
   - Sets `bpd_calculation_complete = true` on last batch

2. **Security Measures:**
   - Same account ownership validation as trigger_big_pay_day
   - PDA derivation verification for each stake
   - Proper Anchor deserialization (no hardcoded byte offsets)

3. **Eligibility Logic:**
   - Must be active stake
   - Must have been created during claim period (start_slot >= claim_config.start_slot && start_slot <= claim_config.end_slot)
   - Must have staked for at least 1 day during claim period
   - Calculates share-days identically to trigger_big_pay_day

4. **Edge Case Handling:**
   - If unclaimed_amount == 0: Sets calculation_complete=true, rate=0, returns
   - If last batch with no eligible stakes (first batch): Resets pagination state to allow retry, keeps pool pending
   - If last batch with no eligible stakes (continuation): Marks calculation complete

5. **Registration:**
   - Added to `mod.rs` module exports
   - Added instruction handler to `lib.rs` program module

## Two-Phase BPD Distribution Workflow

This plan implements **Phase 1** of the two-phase BPD distribution:

### Phase 1: finalize_bpd_calculation (This Plan)
1. Anyone calls `finalize_bpd_calculation` after claim period ends
2. Instruction scans ALL stakes across multiple paginated calls
3. Accumulates total share-days across all eligible stakes
4. Calculates global rate: `helix_per_share_day = (unclaimed * PRECISION) / total_share_days`
5. Stores rate in `ClaimConfig.bpd_helix_per_share_day`
6. Sets `ClaimConfig.bpd_calculation_complete = true`

### Phase 2: trigger_big_pay_day (Plan 02)
1. Will require `bpd_calculation_complete == true` before distributing
2. Will use stored `bpd_helix_per_share_day` for ALL batches
3. Will mark stakes with `bpd_claim_period_id` to prevent duplicate distributions

## Technical Notes

### Pagination Strategy
- Both finalize_bpd_calculation and trigger_big_pay_day use same batch size (20 stakes)
- `bpd_total_share_days` accumulates across finalize batches
- `bpd_remaining_unclaimed` calculated on first finalize batch
- Last batch detected by: `ctx.remaining_accounts.len() < MAX_STAKES_PER_FINALIZE`

### Rate Calculation Precision
- Uses `PRECISION` constant for fixed-point math
- Rate stored as u128 to prevent overflow with large distributions
- Same precision as existing share rate calculations

### Read-Only Philosophy
- finalize_bpd_calculation NEVER writes to StakeAccount
- Only reads stake data for share-days calculation
- All state updates confined to ClaimConfig
- Enables parallel scanning without locking stakes
