## Critical Security Properties Verified

### CRIT-1: First-Batch-Drains-Pool Attack Prevention

**Attack Vector (OLD CODE):**
```rust
// Old code in trigger_big_pay_day (commit 0494e1d, line 149)
let helix_per_share_day = (unclaimed_amount as u128)
    .checked_mul(PRECISION as u128)?
    .checked_div(total_eligible_share_days)?;
```
This calculated rate using ONLY the share-days from the current batch (`total_eligible_share_days`), not the global total. First batch would get a higher rate and drain the pool.

**Fix (NEW CODE):**
```rust
// finalize_bpd_calculation.rs (lines 157-164)
// Accumulates ALL share-days across ALL batches BEFORE calculating rate
claim_config.bpd_total_share_days += batch_share_days; // Each batch
// On LAST batch:
let helix_per_share_day = (unclaimed_amount as u128)
    .checked_mul(PRECISION as u128)?
    .checked_div(claim_config.bpd_total_share_days)?; // GLOBAL total
claim_config.bpd_helix_per_share_day = helix_per_share_day;

// trigger_big_pay_day.rs (line 52)
let helix_per_share_day = claim_config.bpd_helix_per_share_day; // Read pre-calculated
```

### Two-Phase Distribution Workflow

**Phase 1: finalize_bpd_calculation**
1. Anyone calls after claim period ends
2. Processes stakes in batches of 20 (MAX_STAKES_PER_FINALIZE)
3. Accumulates share-days across ALL batches into bpd_total_share_days
4. On last batch: calculates global rate = (unclaimed * PRECISION) / total_share_days
5. Stores rate in bpd_helix_per_share_day
6. Sets bpd_calculation_complete = true

**Phase 2: trigger_big_pay_day**
1. Requires bpd_calculation_complete = true (constraint)
2. Reads pre-calculated helix_per_share_day from ClaimConfig
3. Distributes to stakes in batches of 20 (MAX_STAKES_PER_BPD)
4. Checks bpd_claim_period_id before distributing (skip if already received)
5. Sets bpd_claim_period_id after distributing (mark as received)

**Verification:**
- ✓ Both instructions implemented and registered
- ✓ Constraint enforces ordering (line 29 of trigger_big_pay_day.rs)
- ✓ State fields support workflow
- ✓ Tests verify two-phase flow (all tests call finalizeBpd before triggerBigPayDay)

**Status:** ✓ VERIFIED — Two-phase workflow correctly implemented
