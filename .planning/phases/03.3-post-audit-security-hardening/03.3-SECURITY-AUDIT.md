# Post-Phase 3.2 Security Audit Report

**Date:** 2026-02-08
**Scope:** Full helix-staking Anchor program (all instructions, state, math)
**Auditors:** 7-agent security audit team
**Codebase version:** Post-Phase 3.2 (commit d44b226)
**Verdict:** CONDITIONAL PASS -- 1 CRITICAL, 2 HIGH, 8 MEDIUM vulnerabilities must be resolved before Phase 4

---

## Executive Summary

A 7-agent security audit team independently reviewed the helix-staking Anchor program after the Phase 3.2 BPD security fixes (CRIT-1 and CRIT-2). Each auditor examined the full instruction set, state management, arithmetic safety, access control, and CPI patterns.

**Overall assessment:** The Phase 3.2 fixes (two-phase BPD, duplicate prevention) are correctly implemented. However, the audit uncovered a new CRITICAL vulnerability in the permissionless `finalize_bpd_calculation` design, two HIGH-severity issues (one pre-existing), and eight MEDIUM findings across arithmetic safety, state management, and access control.

**Findings summary:**
| Severity | Count | Status |
|----------|-------|--------|
| CRITICAL | 1 | NEW -- requires fix |
| HIGH | 2 | NEW -- require fix |
| MEDIUM | 8 | NEW -- require fix |
| LOW/INFO | 3 | Noted |
| CONFIRMED FIXED | 3 | Previously reported, verified resolved |

**Clean areas (no issues found):**
- CPI safety and reentrancy patterns (Auditor #5)
- PDA derivation correctness (Auditor #1)
- Signer/authority requirements on gated instructions (Auditor #4)
- Account LEN calculations match field layouts (Auditor #7)
- Monotonic counter consistency across create/unstake/claim (Auditor #7)

## CRITICAL Findings

### CRIT-NEW-1: Permissionless finalize_bpd_calculation Allows BPD Rate Manipulation

**Severity:** CRITICAL
**Corroborated by:** Auditors #1, #2, #3, #4, #7 (5 of 7)
**Files:**
- `finalize_bpd_calculation.rs:12-14` (permissionless caller)
- `finalize_bpd_calculation.rs:151-164` (last-batch rate lock)
- `trigger_big_pay_day.rs:236-241` (distribution uses manipulated rate)

**Description:**
The `finalize_bpd_calculation` instruction is permissionless -- any caller can invoke it and controls which `StakeAccount`s are passed as `remaining_accounts`. The "last batch" signal is `remaining_accounts.len() < MAX_STAKES_PER_FINALIZE` (i.e., fewer than 20 accounts). This creates two attack vectors:

1. **Selective inclusion:** An attacker sends only their own stakes as `remaining_accounts` and marks the batch as "last" by sending fewer than 20. The `bpd_total_share_days` denominator only reflects the attacker's stakes, inflating `bpd_helix_per_share_day` for everyone (but the attacker controls when to lock the rate).

2. **No duplicate prevention in finalize:** Unlike `trigger_big_pay_day` (which checks `bpd_claim_period_id`), `finalize_bpd_calculation` has no mechanism to prevent the same stake from being counted multiple times across batches. An attacker can submit the same stake across multiple finalize calls, inflating the denominator in a controlled way.

**Impact:** Complete manipulation of the BPD distribution rate. An attacker can inflate their own BPD share arbitrarily by selectively including/excluding stakes during the finalize phase.

**Recommended fix (two options):**
- **Option A (simpler):** Make `finalize_bpd_calculation` authority-gated (only protocol authority can call). This is acceptable since BPD is a one-time event per claim period.
- **Option B (decentralized):** Add per-stake tracking (e.g., a `bpd_finalize_period_id` field) to prevent duplicate counting in finalize, and require ALL active eligible stakes to be scanned before rate lock (verified via `global_state.total_stakes_created` counter).

## HIGH Findings

### HIGH-1: crank_distribution u64 Overflow Bricks Rewards at ~50K Staked Tokens

**Severity:** HIGH
**Reported by:** Auditor #6
**File:** `crank_distribution.rs:86-90`

**Description:**
The annual inflation calculation uses a direct `u64.checked_mul()`:

```rust
let annual_inflation = total_staked
    .checked_mul(global_state.annual_inflation_bp)  // u64 * u64
    .ok_or(HelixError::Overflow)?
    .checked_div(100_000_000)
    .ok_or(HelixError::Overflow)?;
```

With `annual_inflation_bp = 3_690_000` and token decimals = 8:
- `total_staked` is in base units (1 HELIX = 100,000,000 base units)
- Overflow occurs when: `total_staked * 3,690,000 > u64::MAX`
- `u64::MAX / 3,690,000 = 4,998,348,841,923` base units
- `4,998,348,841,923 / 100,000,000 = 49,983.49` HELIX tokens

**Impact:** Once total staked exceeds ~49,984 HELIX tokens, `crank_distribution` permanently reverts with `Overflow`. No inflation rewards can be distributed. This is a realistic threshold that would be hit early in protocol life.

**Recommended fix:** Replace the `checked_mul` / `checked_div` sequence with `mul_div(total_staked, global_state.annual_inflation_bp, 100_000_000)` which uses u128 intermediates. The existing `mul_div` helper in `math.rs` already handles this pattern correctly.

### HIGH-2: Unstake Between Finalize and Trigger Causes Permanent Token Loss

**Severity:** HIGH
**Reported by:** Auditor #3
**Files:**
- `trigger_big_pay_day.rs:121` (skips inactive stakes)
- `unstake.rs:114` (sets `is_active = false`)
- `finalize_bpd_calculation.rs:113` (counts active stakes in denominator)

**Description:**
The two-phase BPD flow has a time window between `finalize_bpd_calculation` and `trigger_big_pay_day`. During this window:

1. **Finalize phase:** Stake X is active and counted in `bpd_total_share_days` (the denominator).
2. **Between phases:** Stake X's owner calls `unstake`, setting `is_active = false`.
3. **Trigger phase:** `trigger_big_pay_day` checks `if !stake.is_active { continue; }` and skips Stake X.

Result: Stake X's share-days are in the denominator (reducing the rate for everyone), but Stake X never receives its proportional BPD distribution. That portion of the unclaimed pool is never distributed and is effectively lost (zeroed when `bpd_remaining_unclaimed` is set to 0 on completion).

**Impact:** Permanent loss of unclaimed tokens proportional to the unstaked share-days. In an adversarial scenario, this could be exploited to grief other stakers.

**Recommended fix:** Either:
- Prevent unstaking while BPD is in progress (`bpd_calculation_complete && !big_pay_day_complete` window), or
- Track finalize-phase share-days per stake and allow distribution to inactive stakes that were counted during finalize.

## MEDIUM Findings

### MED-1: Unchecked `as u64` Truncation in BPD Bonus Calculation

**Severity:** MEDIUM
**Reported by:** Auditor #6
**File:** `trigger_big_pay_day.rs:198`

**Description:**
```rust
let bonus = ((*share_days)
    .checked_mul(helix_per_share_day)
    .ok_or(HelixError::Overflow)?
    .checked_div(PRECISION as u128)
    .ok_or(HelixError::DivisionByZero)?) as u64;
```

The `as u64` cast silently truncates if the u128 result exceeds `u64::MAX`. While unlikely with current parameters, this violates the arithmetic safety invariant established in Phase 2.1 where all other u128-to-u64 conversions use `u64::try_from()`.

**Recommended fix:** Replace `as u64` with `u64::try_from(...).map_err(|_| HelixError::Overflow)?`

### MED-2: withdraw_vested u64 Overflow at ~28K HELIX Claims

**Severity:** MEDIUM
**Reported by:** Auditor #6
**File:** `withdraw_vested.rs:163-167`

**Description:**
```rust
let unlocked_vesting = vesting_portion
    .checked_mul(elapsed)        // u64 * u64
    .ok_or(HelixError::Overflow)?
    .checked_div(vesting_duration)
    .ok_or(HelixError::DivisionByZero)?;
```

`vesting_portion` is 90% of `claimed_amount` (in base units). `elapsed` is in slots. With `VESTING_DAYS=30` and `slots_per_day=216,000`:
- `vesting_duration = 30 * 216,000 = 6,480,000` slots
- Overflow when: `vesting_portion * elapsed > u64::MAX`
- Near end of vesting: `elapsed ~ 6,480,000`
- Max safe `vesting_portion = u64::MAX / 6,480,000 = 2,845,640,109,391` base units
- As HELIX tokens: `2,845,640,109,391 / 100,000,000 = 28,456` HELIX

**Impact:** Claims of more than ~28K HELIX tokens will overflow during vesting withdrawal. The transaction reverts (funds are not lost but cannot be withdrawn mid-vesting).

**Recommended fix:** Replace with `mul_div(vesting_portion, elapsed, vesting_duration)` from `math.rs`.

### MED-3: saturating_sub Masks Over-Distribution in trigger_big_pay_day

**Severity:** MEDIUM
**Reported by:** Auditors #3, #6, #7
**File:** `trigger_big_pay_day.rs:229-230`

**Description:**
```rust
claim_config.bpd_remaining_unclaimed = claim_config.bpd_remaining_unclaimed
    .saturating_sub(batch_distributed);
```

If `batch_distributed` exceeds `bpd_remaining_unclaimed` (possible due to rounding in per-stake calculations), `saturating_sub` silently produces 0 instead of flagging the over-distribution. This masks a potential accounting error.

**Recommended fix:** Use `checked_sub` with a specific error, or at minimum emit a warning event when saturation occurs.

### MED-4: Zero-Eligible-Stakes Path Creates Infinite Finalize Loop

**Severity:** MEDIUM
**Reported by:** Auditors #3, #5, #7
**File:** `finalize_bpd_calculation.rs:165-171`

**Description:**
When all batches are processed but no eligible stakes are found (`bpd_total_share_days == 0` after the last batch), the code enters the else branch:

```rust
} else {
    // No eligible stakes found at all - reset pagination state
    claim_config.bpd_remaining_unclaimed = 0;
    claim_config.bpd_total_share_days = 0;
    // DO NOT set calculation_complete
}
```

This resets `bpd_remaining_unclaimed` and `bpd_total_share_days` to 0, which makes `is_first_batch` true again on the next call. The finalize instruction can be called infinitely without ever completing, consuming compute and rent but never progressing.

**Recommended fix:** Set `bpd_calculation_complete = true` and `bpd_helix_per_share_day = 0` when zero eligible stakes are found on the last batch. The trigger phase already handles `helix_per_share_day == 0` correctly (marks complete and returns).

### MED-5: claim_period_id=0 Collision with Default bpd_claim_period_id

**Severity:** MEDIUM
**Reported by:** Auditors #1, #7
**File:** `initialize_claim_period.rs:40` (parameter), `stake_account.rs:33` (default)

**Description:**
`claim_period_id` is passed as an argument to `initialize_claim_period` with no validation. The `bpd_claim_period_id` field on `StakeAccount` defaults to 0 (via Anchor `init` zeroing). If an admin passes `claim_period_id = 0`, the duplicate prevention check in `trigger_big_pay_day`:

```rust
if stake.bpd_claim_period_id == claim_config.claim_period_id {
    continue; // Already received BPD this period
}
```

...will skip ALL stakes that have never received BPD (whose `bpd_claim_period_id` is still the default 0). This effectively prevents any BPD distribution for that claim period.

**Recommended fix:** Add `require!(claim_period_id > 0, HelixError::InvalidParameter)` in `initialize_claim_period`.

### MED-6: admin_mint CEI Violation (State Update After CPI)

**Severity:** MEDIUM
**Reported by:** Auditor #5
**File:** `admin_mint.rs:66-80`

**Description:**
The `admin_mint` instruction performs the CPI `mint_to` call (line 66-77) BEFORE updating the state counter (line 80: `global_state.total_admin_minted = new_total`). This is a Check-Effects-Interactions (CEI) violation:

```rust
// Mint tokens to recipient (INTERACTION first)
token_2022::mint_to(...)?;        // line 66-77

// Update admin mint counter (EFFECT after)
global_state.total_admin_minted = new_total;  // line 80
```

In Solana's single-transaction model, reentrancy is prevented by the runtime (CPI cannot re-enter the same program in most cases). However, CEI violations are a code quality issue and potential risk if future Solana runtime changes relax this guarantee.

**Recommended fix:** Move `global_state.total_admin_minted = new_total` before the `mint_to` CPI call.

---

### MED-7: No Authority Transfer Mechanism

**Severity:** MEDIUM
**Reported by:** Auditor #4
**File:** `global_state.rs:6` (authority field)

**Description:**
`GlobalState.authority` is set at initialization and there is no instruction to transfer it. If the authority key is compromised or needs rotation (e.g., migrating to a multisig), the protocol cannot update it without a program upgrade that manually patches account data.

**Recommended fix:** Add a `transfer_authority` instruction with a two-step pattern (propose + accept) to allow safe authority rotation.

---

### MED-8: Old-Format Stakes (92 bytes) Silently Excluded from BPD

**Severity:** MEDIUM
**Reported by:** Auditors #1, #7
**Files:**
- `finalize_bpd_calculation.rs:87-89` (size check)
- `trigger_big_pay_day.rs:86-88` (size check)
- `stake_account.rs:38` (`OLD_LEN = 92`)

**Description:**
Both `finalize_bpd_calculation` and `trigger_big_pay_day` skip accounts where `data.len() < StakeAccount::LEN` (113 bytes). Stakes created before Phase 3 have the old 92-byte format (`StakeAccount::OLD_LEN`). These stakes are silently excluded from BPD calculation and distribution without any error or event.

While `claim_rewards` performs lazy migration (realloc from 92 to 113 bytes), a user who has not claimed rewards since the migration will have a 92-byte account and be excluded from BPD even if otherwise eligible.

**Recommended fix:** Either:
- Document that users must call `claim_rewards` once to migrate before being BPD-eligible, or
- Handle old-format stakes in finalize/trigger by checking `OLD_LEN` and treating missing fields as defaults, or
- Add a dedicated `migrate_stake` instruction

## LOW / INFO Findings

### LOW-1: Event Field Truncation in BigPayDayDistributed

**File:** `trigger_big_pay_day.rs:259-260`

```rust
total_eligible_share_days: claim_config.bpd_total_share_days.min(u64::MAX as u128) as u64,
helix_per_share_day: helix_per_share_day.min(u64::MAX as u128) as u64,
```

u128 values are truncated to u64 for event emission. This is informational only (events are for indexers, not security-critical), but indexers should be aware that values may be capped.

### LOW-2: Missing Slot Validation in crank_distribution

**File:** `crank_distribution.rs:46-50`

No validation that `clock.slot > global_state.init_slot`. While `get_current_day` would underflow (caught by `checked_sub`), a more descriptive error would improve debuggability.

### INFO-1: Unused `bpd_eligible` Field

**File:** `stake_account.rs:28`

The `bpd_eligible` field is set during `create_stake` but never checked during finalize or trigger (eligibility is determined by `start_slot` range instead). This is wasted account space (1 byte).

## Confirmed Fixed (Previously Reported)

### CRIT-1: BPD Rate Calculated Per-Batch Instead of Globally -- FIXED

**Fixed in:** Phase 3.2, Plan 03.2-01
**Verification:** `finalize_bpd_calculation` now pre-calculates rate across all batches before `trigger_big_pay_day` distributes. Rate is locked in `ClaimConfig.bpd_helix_per_share_day` after finalize completes.

### CRIT-2: Same Stake Can Receive BPD Multiple Times -- FIXED

**Fixed in:** Phase 3.2, Plan 03.2-02
**Verification:** `trigger_big_pay_day` checks `stake.bpd_claim_period_id == claim_config.claim_period_id` and skips already-paid stakes. Field is set after distribution (line 214).

### HIGH-5: remaining_accounts Validation -- FIXED

**Fixed in:** Phase 3.2
**Verification:** Both `finalize_bpd_calculation` and `trigger_big_pay_day` validate account ownership (`account_info.owner != &crate::id()`), proper deserialization, and PDA derivation before processing.
