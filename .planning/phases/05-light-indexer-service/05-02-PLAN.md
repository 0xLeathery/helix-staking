  <files>

  <action>
    1. **src/worker/checkpoint.ts**: Checkpoint management for resumable indexing.
       - `getCheckpoint(programId: string): Promise<{ lastSignature: string | null; lastSlot: number | null }>` -- Query checkpoints table by programId. Return null values if no checkpoint exists (first run).
       - `updateCheckpoint(programId: string, signature: string, slot: number): Promise<void>` -- Upsert checkpoint using Drizzle's `onConflictDoUpdate` on programId. Also increment processedCount. Set updatedAt to new Date().
       - Import `db` from `../db/client.js`, `checkpoints` from `../db/schema.js`, `eq` from `drizzle-orm`.

    2. **src/worker/poller.ts**: Fetch new transaction signatures from RPC.
       - `fetchNewSignatures(rpc, programId, checkpoint): Promise<ConfirmedSignatureInfo[]>` where:
         - `rpc` is the retryable RPC client from lib/rpc
         - `programId` is a string (the program's public key)
         - `checkpoint` is `{ lastSignature: string | null }`
       - Call `rpc.getSignaturesForAddress(new PublicKey(programId), { limit: 1000, until: checkpoint.lastSignature || undefined })`.
         IMPORTANT: Use `until` (not `before`) -- `getSignaturesForAddress` returns newest-first, and `until` means "get signatures until (but not including) this one", which gives us everything AFTER our checkpoint.
       - If no checkpoint exists (first run), fetch the most recent 1000 signatures (the default behavior).
       - Return signatures in REVERSE order (oldest first) so they're processed chronologically.
       - Log signature count at info level, empty results at debug level.

    3. **src/worker/decoder.ts**: Fetch transaction and decode Anchor events.
       - `decodeEventsFromSignature(rpc, signature: string): Promise<Array<{ name: string; data: any; slot: number }>>` where:
         - Calls `rpc.getParsedTransaction(signature, { commitment: 'confirmed', maxSupportedTransactionVersion: 0 })`
         - If transaction is null (pruned/not found), log a warning and return empty array
         - If transaction has no meta or no logMessages, return empty array
         - Extract `tx.meta.logMessages` and pass to `parseEventsFromLogs()` from `../lib/anchor.js`
         - Attach the transaction's `slot` to each decoded event
         - Return array of `{ name, data, slot }`
       - Handle errors gracefully: log and return empty array (don't crash the worker for one bad transaction).
  </action>

  <action>
    1. **src/worker/processor.ts**: Route decoded events to correct database tables.
       - `processEvent(event: { name: string; data: any; slot: number }, signature: string): Promise<void>`
       - Use a switch/map on `event.name` to insert into the correct table:
         - `ProtocolInitialized` -> `protocolInitializedEvents` table
         - `StakeCreated` -> `stakeCreatedEvents` table
         - `StakeEnded` -> `stakeEndedEvents` table
         - `RewardsClaimed` -> `rewardsClaimedEvents` table
         - `InflationDistributed` -> `inflationDistributedEvents` table
         - `AdminMinted` -> `adminMintedEvents` table
         - `ClaimPeriodStarted` -> `claimPeriodStartedEvents` table
         - `TokensClaimed` -> `tokensClaimedEvents` table
         - `VestedTokensWithdrawn` -> `vestedTokensWithdrawnEvents` table
         - `ClaimPeriodEnded` -> `claimPeriodEndedEvents` table
         - `BigPayDayDistributed` -> `bigPayDayDistributedEvents` table
       - For each insert, include `signature` and `slot` along with event-specific fields.
       - Convert Anchor's BN values to strings for text columns: `event.data.amount.toString()`, `event.data.tShares.toString()`, etc. Convert Pubkey values with `.toString()` or `.toBase58()`.
       - Use `onConflictDoNothing()` on signature (unique constraint) to handle duplicate processing gracefully -- if the same signature is processed twice, silently skip it.
       - For `InflationDistributed` events, add gap detection: after inserting, query the previous distribution day from the table. If `event.data.day - previousDay > event.data.daysElapsed`, log a WARNING about potential missed events. This is informational only (no action, just alerting).
       - Log each processed event at debug level, and errors at error level (with signature for investigation).

    2. **src/worker/index.ts**: Worker entrypoint with polling loop and graceful shutdown.

       **Startup:**
       - Import and call env validation (from `../lib/env.js`) at top level to fail fast on missing config.
       - Create RPC client from `env.RPC_URL`.
       - Log startup info: programId, pollInterval, rpcUrl (masked).

       **Polling loop:**
       - Use `setInterval` with `env.POLL_INTERVAL_MS` (default 5000ms).
       - Each tick:
         1. Get current checkpoint via `getCheckpoint(env.PROGRAM_ID)`
         2. Fetch new signatures via `fetchNewSignatures(rpc, env.PROGRAM_ID, checkpoint)`
         3. If no new signatures, return early (debug log)
         4. For each signature (oldest first):
            a. Decode events via `decodeEventsFromSignature(rpc, signature)`
            b. For each event: `processEvent(event, signature)`
            c. Update checkpoint with this signature + slot
         5. Log batch summary: N signatures processed, M events stored
       - Wrap entire tick in try/catch: log errors at error level, do NOT crash. Next tick will retry.
       - Track `isProcessing` flag to prevent overlapping ticks (if processing takes longer than interval).

       **Graceful shutdown:**
       - Register handlers for SIGTERM and SIGINT.
       - On signal: set `isShuttingDown = true`, clear the polling interval.
       - Wait for current batch to finish (if `isProcessing`, wait up to 30s).
       - Close database pool via `pool.end()`.
       - Log "Graceful shutdown complete" and `process.exit(0)`.
       - If shutdown times out after 30s, log error and `process.exit(1)`.
       - Prevent double-shutdown with `isShuttingDown` flag.

       **Important details:**
       - Process signatures one at a time (not in parallel) to maintain ordering and checkpoint consistency.
       - If any single signature fails to decode/process, log the error and continue to next signature (don't abort the batch).
       - Update checkpoint after EACH signature (not at end of batch) so crash recovery is granular.
  </action>
